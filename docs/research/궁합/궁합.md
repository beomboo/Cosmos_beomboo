# 궁합(宮合)

_최종 확인: 2026-07-18_

**현황**: 미구현. 02_경쟁앱 분석·05_결과화면 조사에서 "궁합 프리미엄 리포트 22,000원" 같은 결제 사례는 확인됐으나, 궁합 자체의 계산 방식·우리 앱과의 기술적 관계는 별도로 조사됨. 구현 여부는 사람 결정 대기.

## 개념
두 사람의 사주팔자를 나란히 놓고 오행 조화·일주(日柱)의 합충(合沖) 관계를 분석 — 좋은 궁합 = 오행이 같은 게 아니라 **서로 부족한 오행을 채워주는 상보(相補) 관계**.

- **겉궁합 vs 속궁합**: 겉궁합은 **띠(년지)만 비교**(생년만 있으면 계산 가능, 정확도 낮음). 속궁합은 **사주 8글자 전체** 비교(생년월일시 전부 필요).
- **판단 핵심 요소 3가지**: (1) 오행 보완 관계, (2) 천간합(天干合) — 갑기·을경·병신·정임·무계 5쌍, (3) 일지의 합·충·형·파.

## 우리 앱과의 기술적 관계 — "입력이 2인분인 확장"에 가까움 (택일처럼 완전히 다른 제품은 아님)

- **오행 보완 관계**: 이미 있는 `FourPillars.ohaengCount`를 두 사람 것 각각 계산해서 비교하면 됨 — 새 계산 로직이 거의 필요 없는 가장 쉬운 부분.
- **천간합**: `stemIndex`끼리 5쌍 조합표(인덱스 차이가 항상 5)만 있으면 되는 고정 로직 — 십신·십이운성과 같은 급의 난이도.
- **일지 합충형파**: 우리 코드베이스에 전혀 없는 새 개념. `saju`(Dart) 패키지의 `core/relations.dart`(495줄)가 정확히 이 부분을 담당 — 조견표는 그대로 재사용 가능하지만, **`analyzeRelations(FourPillars pillars)`는 인자가 하나뿐인 "한 사람의 사주 내부 관계" 분석 함수라 궁합(2인 비교)용이 아니다.** 조견표(어느 지지끼리 합/충/형/파/해인지)는 재사용하되, "사람 A의 지지 × 사람 B의 지지"를 이 표로 검사하는 비교 함수는 새로 작성해야 한다.
- **입력 모델**: 택일(완전히 다른 제품, [../사주팔자/택일.md](../사주팔자/택일.md))과 달리, 궁합은 여전히 "명식 해석"이라는 같은 기본 틀 안에 있고 입력받는 사람이 1명→2명으로 늘어날 뿐이다 — `BirthInfo`를 두 번 받아 `calculateFourPillars()`를 두 번 호출하고 비교하는 층 하나만 얹으면 되는 구조.

## 합충형파해 조견표 (`saju`(Dart) `core/relations.dart` 원문)

| 관계 종류 | 조견표 |
|---|---|
| **천간합(天干合)** 5쌍 | 갑기=토, 을경=금, 병신=수, 정임=목, 무계=화 (化하는 오행까지 포함) |
| **육합(六合)** 6쌍 | 자축=토, 인해=목, 묘술=화, 진유=금, 사신=수, 오미=토 |
| **삼합(三合)** 4그룹 | 인오술=화, 신자진=수, 해묘미=목, 사유축=금 (12신살과 동일 그룹) |
| **방합(方合)** 4그룹 | 인묘진=목, 사오미=화, 신유술=금, 해자축=수 (계절별 인접 지지, 삼합과 다름) |
| **충(沖)** 6쌍 | 자오, 축미, 인신, 묘유, 진술, 사해 |
| **해(害)** 6쌍 | 자미, 축오, 인사, 묘진, 신해, 유술 — [신살.md](../사주팔자/신살.md)의 육해살 계산 기준이 바로 이 표 |
| **파(破)** 6쌍 | 자유, 축진, 인해, 묘오, 사신, 미술 |
| **형(刑)** | 인사신(무은지형)·축술미(지세지형)·자묘(무례지형)·진진/오오/유유/해해(자형) |

### 파생 콘텐츠 — "사주 내 합충형파해"(자체 진단, 2026-07-18 재탐색 발견)
위 조견표는 원래 궁합(2인 비교)용으로 정리했지만, `saju`(Dart)의 `analyzeRelations(FourPillars pillars)` 자체가 애초에 **인자가 한 사람 것 하나뿐인 "자기 사주 내부 관계" 분석 함수**라는 점을 다시 보면, 상대방 없이 **자기 사주 4기둥(연/월/일/시)의 지지끼리 이 표로 대조**하는 것만으로도 독립된 콘텐츠가 된다 — 예: "내 사주는 일지와 시지가 충(沖) 관계라 이사·이동이 잦은 편" 같은 자가진단. 웹 검색으로 "내 사주 충이 있다면 어떻게 해야 할까?"류 YouTube 콘텐츠가 궁합과 무관하게 독립적으로 소비되는 것을 확인해, 이를 [기초개념.md](../사주팔자/기초개념.md)의 "콘텐츠 갭 종합" 표에도 별도 행으로 추가했다. 계산 난이도는 궁합보다 오히려 낮다 — 두 번째 사람 입력·다중 프로필 저장 구조가 전혀 필요 없이 이미 계산된 자기 사주 4기둥끼리만 대조하면 되기 때문(궁합의 "저장 UX" 관련 논의는 이 파생 콘텐츠에는 해당하지 않음).

## 실제 구현 시 참고할 시장 관행
- **저장 구조**: 실제 앱들은 "결과 페이지에서 사용자를 무제한 추가"하는 다중 프로필 저장 방식을 씀 — 우리 `BirthInfoStore`는 현재 1명(본인) 전용이라, 궁합을 구현한다면 **저장소 자체를 "여러 명 저장 가능"으로 확장**해야 함(단순히 입력 필드 2개 추가가 아님).
- **시간 입력**: 시간 없이 년/월/일 3기둥만으로 궁합을 기본 제공하고, 시간 추가 시 시주까지 반영해 정밀해지는 방식이 흔함 — 우리 앱의 기존 "태어난 시간 모름" 처리 패턴과 그대로 맞음.
- **결과 표현**: 딱딱한 합충형파해 분석 나열 대신 **100점 만점 점수 + 5단계 등급**(천생연분/좋은 궁합/보통/주의 필요/어려운 궁합)으로 압축해 보여주는 게 흔한 성공 패턴 — [../사주팔자/신살.md](../사주팔자/신살.md)의 "체크리스트+자가진단" 포맷과 같은 "복잡한 계산 → 단순한 점수/등급 번역" 원리.
- **겉궁합의 실제 형태**: 독립된 "MVP 화면"이라기보다 **"오늘의 띠별 궁합" 같은 매일 갱신되는 반복 콘텐츠**로 구현되는 경우가 많다(네이트 운세 등) — "속궁합의 축소판"이 아니라 "오늘의 운세류"에 더 가까운 콘텐츠로 분류하는 게 정확함.
- **가격대**: 실제 시장에서 궁합은 고가 유료 콘텐츠(22,000원 사례 확인)로 취급되는 경우가 많음 — "결제 로직" 사람 결정과 자연스럽게 묶이는 기능.
- **두 번째 사람의 입력 UX — "캐주얼 vs 전문가용"이 아니라 "단발성 웹 계산기 vs 설치형 앱"이 진짜 경계선(2026-07-17 재확인·정정, 구현 여부는 여전히 사람 결정 대기)**:
  - **① 단발성 웹 계산기형** — 데이사주(`daysaju.com/gunghap/saju`)처럼 "궁합 계산기" 단일 페이지를 제공하는 서비스는 나·상대 섹션이 나란히 있고 **매번 새로 입력**하는 폼만 존재. 성별·이름·생년월일·태어난 시간(모름 옵션 있음)·양력/음력을 그때그때 채우며, 저장된 프로필을 고르는 기능은 없음. 페이지 상단에 로그인 버튼은 있지만 궁합 계산 자체는 **비회원도 사용 가능** — 로그인은 다른 부가 기능(히스토리 등)용으로 보임.
  - **② 설치형 앱형(전문 만세력 앱 + 대중 운세 앱 공통)** — 원광만세력·만세력 천을귀인·8자어때 같은 역술인·헤비유저 대상 만세력 앱은 물론, **국내 최대 대중 운세 앱인 포스텔러(860만 다운로드)도 동일하게 "프로필과 즐겨찾기, 그룹으로 정리해 효율적으로 관리"하는 저장형 UX를 채택**하고 있음을 2026-07-17 재조사로 확인 — 즉 "저장된 프로필 선택" 패턴은 전문/헤비유저 앱만의 것이 아니라 **설치형 앱이라면 대중 캐주얼 앱도 일반적으로 채택하는 표준**이다. 원광만세력은 이름·메모·생년월일시·성별을 붙여 명조를 저장·검색·불러오기·삭제할 수 있는 목록을 제공(과거 라이트 버전은 50명 제한, 정식 버전은 무제한). 만세력 천을귀인은 이 목록에서 "본인은 짧게 터치, 상대방은 길게 눌러" 선택하면 바로 궁합 분석이 시작되는 제스처 UX를 씀. 8자어때는 저장한 명조를 가족·친구 같은 그룹으로 묶어 정리하고 이름·생년월일·나이로 검색할 수 있게 함. 포스텔러는 결과 화면에서 저장 버튼으로 명식을 보관해두고(사용자 후기에 "가족 사주를 전부 저장해뒀다"는 사례 확인) 프로필·즐겨찾기·그룹 단위로 관리하는 구조로 보이나, 공개된 자료만으로는 관계 라벨(가족/친구/연인) 명칭 체계나 최대 저장 인원 등 세부 UI까지는 확인하지 못함 — 다만 "궁금한 사람들 하나하나 입력하고 저장하는 게 힘들다"는 사용자 불만도 함께 확인돼, 저장 UX 자체는 있어도 등록 과정(입력 폼)이 여전히 번거롭다는 실사용 피드백이 존재함. 점신·헬로우봇(대중 운세 앱 상위권)은 궁합에 두 사람 생년월일시가 필요하다는 점은 확인됐으나, 프로필 저장·관리 UX 세부사항은 공개 자료(웹 검색·앱스토어 설명)만으로 확인하지 못함(JS 렌더링 사이트 접근 제약).
  - **로그인 필요 여부**: 전문 만세력 앱들은 공통적으로 **로그인 없이 기기 내부(로컬) 저장**을 채택 — 만세력 천을귀인은 앱 설명에 "개인정보를 수집하지 않으며 저장 데이터는 기기 내부에만 보관"한다고 명시. 원광만세력도 명조 저장·검색 기능 자체는 계정 시스템과 무관(앱 다운로드에 스토어 계정이 필요한 것과는 별개). 즉 **"다중 프로필 저장"과 "회원가입"은 별개 문제** — 로그인 없이도 로컬 DB에 여러 사람을 저장하는 게 시장의 일반적 관행이며, 우리 앱처럼 계정 시스템이 없는 구조에서도 채택 가능한 패턴으로 보임(포스텔러 같은 대중 앱은 회원 로그인을 쓰는 것으로 보이나, 저장 기능 자체가 로그인을 반드시 요구하는지는 확인하지 못함).
  - **이름/닉네임 표시(개인정보 배려)**: 조사한 범위에서는 이니셜 처리·관계 라벨만 노출 등 **별도의 익명화 UX는 확인되지 않음** — 위 앱들 모두 단순 자유입력 "이름"/"메모" 필드에 사용자가 원하는 대로(실명이든 별명이든) 적어 넣는 방식. 개인정보 보호는 "로컬 저장 + 서버 미수집"이라는 저장 위치 차원에서 다뤄지고 있고, 표시 문구 차원의 특별한 배려는 시장에서 표준화돼 있지 않은 것으로 판단됨.
  - **우리 앱에 대한 시사점**: 우리 앱은 설치형(웹 계산기가 아님)이므로 위 ②(설치형 앱형) 패턴이 더 적합한 레퍼런스다 — "단발성 계산기라 저장 없이 매번 입력받아도 무방하다"는 판단은 시장 관행과 맞지 않고, 궁합을 구현한다면 처음부터 다중 프로필 저장 구조(관계 라벨 포함)로 설계하는 편이 표준에 가까움.
