# 궁합(宮合)

_최종 확인: 2026-07-25_

**현황**: 미구현. 궁합 자체의 계산 방식·우리 앱과의 기술적 관계, 그리고 "두 번째 사람" 입력 UX의 시장 표준까지 조사 완료. 구현 여부·저장소 구조 변경은 사람 결정 대기(`docs/DECISIONS.md` C-7).

## 개념
두 사람의 사주팔자를 나란히 놓고 오행 조화·일주(日柱)의 합충(合沖) 관계를 분석 — 좋은 궁합 = 오행이 같은 게 아니라 **서로 부족한 오행을 채워주는 상보(相補) 관계**.

- **겉궁합 vs 속궁합**: 겉궁합은 **띠(년지)만 비교**(생년만 있으면 계산 가능, 정확도 낮음). 속궁합은 **사주 8글자 전체** 비교(생년월일시 전부 필요).
- **판단 핵심 요소 3가지**: (1) 오행 보완 관계, (2) 천간합(天干合) — 갑기·을경·병신·정임·무계 5쌍, (3) 일지의 합·충·형·파.

## 우리 앱과의 기술적 관계 — "입력이 2인분인 확장"에 가까움 (택일처럼 완전히 다른 제품은 아님)

- **오행 보완 관계**: 이미 있는 `FourPillars.ohaengCount`를 두 사람 것 각각 계산해서 비교하면 됨 — 새 계산 로직이 거의 필요 없는 가장 쉬운 부분.
- **천간합**: `stemIndex`끼리 5쌍 조합표(인덱스 차이가 항상 5)만 있으면 되는 고정 로직 — 십신·십이운성과 같은 급의 난이도.
- **일지 합충형파**: 우리 코드베이스에 전혀 없는 새 개념. `saju`(Dart) 패키지의 `core/relations.dart`(495줄)가 정확히 이 부분을 담당 — 조견표는 그대로 재사용 가능하지만, **`analyzeRelations(FourPillars pillars)`는 인자가 하나뿐인 "한 사람의 사주 내부 관계" 분석 함수라 궁합(2인 비교)용이 아니다.** 조견표(어느 지지끼리 합/충/형/파/해인지)는 재사용하되, "사람 A의 지지 × 사람 B의 지지"를 이 표로 검사하는 비교 함수는 새로 작성해야 한다.
- **입력 모델**: 택일(완전히 다른 제품, [../사주팔자/택일.md](../사주팔자/택일.md))과 달리, 궁합은 여전히 "명식 해석"이라는 같은 기본 틀 안에 있고 입력받는 사람이 1명→2명으로 늘어날 뿐이다 — `BirthInfo`를 두 번 받아 `calculateFourPillars()`를 두 번 호출하고 비교하는 층 하나만 얹으면 되는 구조.

## 합충형파해 조견표 (`saju`(Dart) `core/relations.dart` 원문)

| 관계 종류 | 조견표 |
|---|---|
| **천간합(天干合)** 5쌍 | 갑기=토, 을경=금, 병신=수, 정임=목, 무계=화 (化하는 오행까지 포함) |
| **육합(六合)** 6쌍 | 자축=토, 인해=목, 묘술=화, 진유=금, 사신=수, 오미=토 |
| **삼합(三合)** 4그룹 | 인오술=화, 신자진=수, 해묘미=목, 사유축=금 (12신살과 동일 그룹) |
| **방합(方合)** 4그룹 | 인묘진=목, 사오미=화, 신유술=금, 해자축=수 (계절별 인접 지지, 삼합과 다름) |
| **충(沖)** 6쌍 | 자오, 축미, 인신, 묘유, 진술, 사해 |
| **해(害)** 6쌍 | 자미, 축오, 인사, 묘진, 신해, 유술 — [신살.md](../사주팔자/신살.md)의 육해살 계산 기준이 바로 이 표 |
| **파(破)** 6쌍 | 자유, 축진, 인해, 묘오, 사신, 미술 |
| **형(刑)** | 인사신(무은지형)·축술미(지세지형)·자묘(무례지형)·진진/오오/유유/해해(자형) |

### 파생 콘텐츠 — "사주 내 합충형파해"(자체 진단, 궁합과 독립된 콘텐츠)
위 조견표는 원래 궁합(2인 비교)용으로 정리했지만, `saju`(Dart)의 `analyzeRelations(FourPillars pillars)` 자체가 애초에 **인자가 한 사람 것 하나뿐인 "자기 사주 내부 관계" 분석 함수**라는 점을 보면, 상대방 없이 **자기 사주 4기둥(연/월/일/시)의 지지끼리 이 표로 대조**하는 것만으로도 독립된 콘텐츠가 된다 — 예: "내 사주는 일지와 시지가 충(沖) 관계라 이사·이동이 잦은 편" 같은 자가진단. "내 사주 충이 있다면 어떻게 해야 할까?"류 YouTube 콘텐츠가 궁합과 무관하게 독립적으로 소비되는 것을 확인해, [기초개념.md](../사주팔자/기초개념.md)의 "콘텐츠 갭 종합" 표에도 별도 행으로 반영했다. 계산 난이도는 궁합보다 낮다 — 두 번째 사람 입력·다중 프로필 저장 구조가 전혀 필요 없이 이미 계산된 자기 사주 4기둥끼리만 대조하면 되기 때문(아래 "두 번째 사람 입력 UX" 논의는 이 파생 콘텐츠에는 해당하지 않음).

## 두 번째 사람("상대방") 입력 UX — 시장 표준 조사

**결론**: 설치형 앱에서는 "매번 직접 입력"이 아니라 "저장된 프로필에서 선택"이 시장 표준이다. 매번 새로 입력하는 방식은 로그인조차 없는 1회성 웹 계산기(데이사주 등)에서만 흔하고, 설치형 앱은 전문 만세력 앱이든 대중 캐주얼 앱이든 예외 없이 프로필 저장 구조를 채택한다. 우리 앱은 설치형(웹 계산기가 아님)이므로 저장 프로필 방식이 표준에 부합한다.

| 구분 | 사례 | 저장 방식 | 최대 인원 | 관계 라벨 | 회원가입 |
|---|---|---|---|---|---|
| 1회성 웹 계산기 | 데이사주(daysaju.com/gunghap/saju) | 저장 없음, 매번 직접 입력 | 해당 없음 | 없음 | 불필요(비회원 사용 가능) |
| 전문 만세력 앱 | 원광만세력 | 이름·메모·생년월일시·성별로 명조 저장·검색·불러오기 | 구버전 "라이트" 무료 50명 / 정식 버전 무제한 | 없음(이름+자유 메모 텍스트) | 불필요(로컬 저장) |
| 전문 만세력 앱 | 만세력 천을귀인 | 저장 목록에서 짧게 터치=본인, 길게 눌러=상대 선택 후 바로 궁합 분석 | 확인 안 됨 | 없음 | 불필요(앱 설명에 "개인정보 미수집·기기 내부 저장" 명시) |
| 전문 만세력 앱 | 8자어때 | 그룹으로 묶어 정리, 이름/생년월일/나이로 검색 | 확인 안 됨 | 그룹 단위(가족/친구 등) | 확인 안 됨 |
| 대중 캐주얼 앱 | 정통사주(누적 200만 다운로드) | 나·가족·연인·친구·자녀로 구분해 등록 | **무제한** | **있음** — 나/가족/연인/친구/자녀 | 불필요(회원가입 없음, 이름+생년월일만 수집, 클라우드 저장 없음) |
| 대중 캐주얼 앱 | 포스텔러(860만 다운로드) | 프로필·즐겨찾기·그룹으로 정리 | 확인 안 됨 | 확인 안 됨(그룹 기능은 있으나 라벨 체계 미공개) | 회원 로그인 사용(저장 기능 자체가 로그인을 요구하는지는 불확실) |

### 세 가지 질문에 대한 답
1. **직접 입력 vs 저장 프로필**: 저장 프로필이 시장 표준. 조사한 설치형 앱 5곳 전부 저장 구조를 쓰고, 매번 직접 입력은 로그인도 없는 1회성 웹 계산기에서만 나타난다.
2. **저장 개수**: 무제한이 대세. 정통사주(대중 캐주얼)와 원광만세력 정식 버전(전문가용) 모두 무제한이며, 유일하게 확인된 숫자 제한은 원광만세력 구버전 "라이트" 무료판의 50명 — 무료/유료를 인원 수로 가르는 선례로 C-10(결제 게이트) 논의에도 참고할 만하다.
3. **관계 라벨**: 존재한다. 정통사주는 나/가족/연인/친구/자녀 5개 라벨을 명시적으로 구분해 등록받고, 8자어때는 가족/친구 등 그룹 단위로 묶는다. 반면 원광만세력·만세력 천을귀인은 라벨 없이 이름+메모 자유입력만 제공 — "관계 라벨"이 모든 앱의 절대 표준까지는 아니지만, 궁합 기능을 핵심으로 내세우는 대중 앱(정통사주)일수록 라벨을 명시적으로 붙이는 경향이 뚜렷하다. 이니셜 처리 등 표시 문구 차원의 개인정보 배려는 시장에서 확인되지 않는다 — 프라이버시는 "로그인 없이 로컬 저장"이라는 저장 위치 차원에서 다뤄진다.

### 우리 앱에 대한 시사점
- `BirthInfoStore`를 다중 프로필 구조로 확장한다면 (a) 저장 인원 제한 없음, (b) 이름 자유입력 + 관계 라벨 선택(나/가족/연인/친구/기타 정도), (c) 로그인 없이 온디바이스 저장 유지가 시장 표준과 맞는다.
- 무료/유료 경계선(C-10)을 "저장 인원 수"로 가르는 방식(무료 N명, 유료 무제한)은 원광만세력이 실제로 썼던 선례가 있어 검토 가치가 있다 — 다만 이는 사업 결정이라 이 리서치에서 확정하지 않는다.
- 회원가입 요구 여부는 사업 모델에 따라 갈릴 수 있다 — 정통사주·원광만세력류(로그인 없음)와 포스텔러류(로그인 있음) 두 갈래가 공존하므로 시장이 어느 한쪽을 강제하지는 않는다.

## 실제 구현 시 참고할 그 외 시장 관행
- **시간 입력**: 시간 없이 년/월/일 3기둥만으로 궁합을 기본 제공하고, 시간 추가 시 시주까지 반영해 정밀해지는 방식이 흔함 — 우리 앱의 기존 "태어난 시간 모름" 처리 패턴과 그대로 맞음.
- **결과 표현**: 딱딱한 합충형파해 분석 나열 대신 **100점 만점 점수 + 5단계 등급**(천생연분/좋은 궁합/보통/주의 필요/어려운 궁합)으로 압축해 보여주는 게 흔한 성공 패턴 — [신살.md](../사주팔자/신살.md)의 "체크리스트+자가진단" 포맷과 같은 "복잡한 계산 → 단순한 점수/등급 번역" 원리.
- **겉궁합의 실제 형태**: 독립된 "MVP 화면"이라기보다 **"오늘의 띠별 궁합" 같은 매일 갱신되는 반복 콘텐츠**로 구현되는 경우가 많다(네이트 운세 등) — "속궁합의 축소판"이 아니라 "오늘의 운세류"에 더 가까운 콘텐츠로 분류하는 게 정확함.
- **가격대**: 실제 시장에서 궁합은 고가 유료 콘텐츠(포스텔러 궁합 프리미엄 리포트 22,000원 사례 확인)로 취급되는 경우가 많음 — 결제 로직(C-10) 사람 결정과 자연스럽게 묶이는 기능.
