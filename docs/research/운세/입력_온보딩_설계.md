## "태어난 시간을 모를 때" 처리 — 실제 앱 4가지 패턴 (2026-07-18 재검증)
기존 문서는 "엄격 요구형/유연 허용형/권장안" 3가지를 추상적으로 제시했으나, 실제 서비스를 찾아 대조한 결과 아래 4가지 구체적 패턴으로 재정리한다. 국내 사주 앱뿐 아니라 해외 서양 점성술 앱·BaZi 계산기도 개념적으로 동일한 선택지(정보 누락분만 생략 vs. 추정치로 채움)로 수렴한다.

| 패턴 | 실제 사례 | 동작 |
|---|---|---|
| **3주 표시형(권장)** | [데이사주](https://daysaju.com/manseryeok) | "모름" 선택 시 시주만 제외한 연주/월주/일주 3개만 계산·표시. "시주가 빠지므로 사주 정보의 75%는 확인 가능"이라고 명시적으로 안내 — 무엇이 빠졌는지 투명하게 밝히는 방식 |
| **Time Unknown 체크박스형** | 서양 점성술 계산기 전반(예: [cafeastrology.com natal chart](https://astro.cafeastrology.com/natal.php)), 해외 [bazi-calculator.app](https://bazi-calculator.app/) | "Time Unknown" 체크 시 어센던트/하우스(서양) 또는 시주(BaZi)만 결과에서 제외하고 나머지는 그대로 계산 — 데이사주와 개념적으로 동일한 국제 공통 패턴. "노정(정오)으로 임의 대입해 확정된 값처럼 보여주면 안 된다"는 것이 실무자들의 공통 권고 |
| **대략적 시간대 유도형** | [포스텔러](https://info.n2i.co.kr/2025/05/blog-post_24.html) | "시간을 정확히 모르시면 대략적인 시간대라도 입력해보세요"라는 문구로 완전 스킵보다 근사 입력을 유도. 이후 시태양시·야자시/조자시·12지 시진법 등 다양한 보정 옵션을 제공해 오차를 흡수 |
| **자동 추천형(주의 필요)** | [기운사](https://giunsa.com/blog/birth-time-unknown) | "모름" 선택 시 랜덤이 아니라 "조후(계절 기후) 균형과 오행 결핍 보완"을 근거로 명리학적으로 그럴듯한 시주를 자동으로 채워 넣음(겨울생→火 기운 시주 배정 등). 사용자에게는 추정치임을 명확히 라벨링하지 않으면 실제 출생시로 오인시킬 위험이 있어 **비권장** — 채택하더라도 "추정 시주"임을 결과 화면에 반드시 병기해야 함 |

**결론**: 기존 문서의 "권장안"(모름 선택 시 3주만 표시 + 넛지 문구)은 데이사주가 실제로 쓰는 방식과 정확히 일치하며, 국제적으로도 "Time Unknown 체크박스형"과 개념이 같은 검증된 패턴이다 — 그대로 유지해도 좋다. 다만 포스텔러처럼 "완전 스킵"보다 "대략적 시간대라도" 입력하도록 먼저 유도하는 중간 단계를 추가하면 이탈 방지에 도움이 될 수 있다. 기운사식 "자동 추천"은 흥미로운 아이디어지만 정확도를 오해시킬 소지가 있어 채택 시 라벨링이 전제조건이다.

## 성별 입력 필수 여부 재검증 (2026-07-18)
- **포스텔러**: 개인정보 입력 2단계에서 "이름, 성별, 생년월일시, 태어난 도시를 차례로 입력"받는 흐름 확인 — 성별이 생년월일시와 동등한 필수 항목으로 취급됨.
- **데이사주**: "만세력 계산기는 생년월일시와 성별만 입력하면 명식표·오행 분석·대운/세운/월운까지 확인 가능" — 대운 계산의 입력값으로 성별이 생년월일시와 나란히 명시됨.
- **해외 BaZi 계산기**([bazi-calculator.app](https://bazi-calculator.app/))도 Year/Month/Day/Hour와 함께 Male/Female 선택을 입력 폼에 포함하며, "Luck Pillar(대운)는 생년월일과 성별로 계산되는 나이에서 시작한다"고 명시.
- **점신**: 홈 화면의 "오늘의 운세"류 가벼운 콘텐츠는 생년월일만으로도 볼 수 있으나, 만세력·대운 등 정밀 사주 콘텐츠에는 별도로 성별이 필요한 구조로 보임(스토어 설명 기준 — 실제 스크린샷 확인은 로그인 세션 제약으로 이번 조사에서 확보하지 못함).

**결론**: 조사한 국내외 앱 어디에서도 "성별을 완전히 스킵 가능한 선택값"으로 두고 대운을 계산하는 사례는 발견하지 못했다 — 성별은 대운 순행/역행(연간 천간의 음양 × 성별 조합)을 결정하는 핵심 입력값이라 생략 시 대운 자체를 계산하지 않거나 명시적으로 숨기는 것이 시장 관행이다. 최근 `saju-developer`가 이번 작업 범위에서 성별 검증을 제외한 결정은 "지금 당장 대운 기능이 없어서"라는 전제라면 타당하지만, 향후 대운 화면을 구현하는 시점에는 성별 필수 검증을 함께 넣어야 시장 표준과 일치한다. 이 판단은 `saju-planner`가 대운 기능 게이트를 열 때 참고할 근거로 남긴다(계산 로직 자체는 [../사주팔자/대운_세운.md](../사주팔자/대운_세운.md) 참고).

## 자시(子時) 관련 UX 안내
23:00~01:00 출생자는 일주가 전날/당일 중 어느 쪽으로 계산되는지 앱마다 다르다 — 이 구간 입력 시 "정확한 결과를 위해 병원 기록상 시간을 확인해보세요" 툴팁 고려. 포스텔러는 실제로 "야자시/조자시" 세분화 옵션과 시태양시·서머타임 자동 보정을 사용자가 직접 선택하게 하는 방식으로 이 문제를 UI에 노출한다 — 관법을 사용자에게 아예 숨기지 않고 "선택 가능한 고급 옵션"으로 노출하는 것도 하나의 실제 사례. 실제 계산 관법 3가지(`midnight`/`splitJasi`/`jasi`)와 우리 코드의 현재 선택은 [../사주팔자/계산_정확도_이슈.md](../사주팔자/계산_정확도_이슈.md) 참고.

## 온보딩 흐름 (경쟁 앱 공통 패턴 — 2026-07-18 재검증)
포스텔러의 실제 개인정보 입력 단계 순서(이름 → 생년월일시 → 성별 → 태어난 도시)를 반영해 갱신(2026-07-18, 이전 버전은 "이름 → 성별 → 생년월일시" 순으로 잘못 요약돼 있어 정정 — 아래 1~4번 목록이 실제 확인된 순서):
1. (선택) 이름/애칭 입력 — 국내 앱 다수가 결과 화면에서 "OOO님의 사주"처럼 이름을 노출하기 위해 받음
2. 생년월일시 입력(달력 UI+시간 스크롤 피커), 양력/음력 토글
3. 성별 선택 — 필수(위 절 참고)
4. (선택) 출생지 입력 — 스킵 가능하게
5. 로딩 애니메이션(계산 중 — 캐릭터/일러스트 연출 넣기 좋은 지점)
6. 결과 화면 진입

로딩 애니메이션·자시 안내 문구 등 세부 연출에 대한 추가 최신 사례는 이번 조사에서 확보하지 못함(스토어 설명·블로그 텍스트 기반 조사의 한계 — 실제 스크린 흐름은 앱 직접 설치 후 확인 필요, `docs/research/README.md`의 조사 방법 한계 참고).

## UX 주의점
- 입력 필드가 많으면 이탈률 상승 → 필수값 최소화+선택값은 "나중에 추가" 유도
- 시간 모름 케이스가 실제로 매우 흔함(병원 기록 없는 경우 다수) → 이 케이스의 결과 품질/설명이 앱 신뢰도에 직결
- 성별은 국내외 시장 전반에서 "생략 가능한 선택값"으로 취급되지 않는다 — UI에서 스킵을 허용하려면 최소한 "대운 정보는 제공되지 않습니다"처럼 그 대가를 명시해야 함
