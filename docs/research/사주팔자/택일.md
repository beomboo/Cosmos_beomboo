# 택일(擇日) — 콘텐츠 갭이 아니라 별도 제품 형태

_최종 확인: 2026-07-15_

**현황**: 미구현. "사주냥" 마케팅 자료에서 신살과 함께 발견. 구현 여부는 사람 결정 대기.

## 정의
"좋은 날을 고른다"는 뜻으로, 결혼·이사·출산(제왕절개 등) 같은 미래 일정에 대해 음양오행·사주·절기·풍수를 종합해 길일을 찾아주는 서비스.

## 왜 다른 콘텐츠 갭(십신·용신·대운/세운·공망·십이운성·신살)과 근본적으로 다른가
그 6가지는 전부 **"이미 정해진 한 사람의 생년월일시(BirthInfo) → 그 명식을 해석"**하는 동일한 입출력 모델을 공유한다. 택일은 **"목적(결혼/이사/출산)+탐색 대상 기간 → 그 기간 안에서 길일 후보를 스캔해 달력으로 보여주기"**로 입력·출력 모델 자체가 다르다(결혼 택일은 두 사람의 명식을 동시에 고려하기도 함). 기존 계산·콘텐츠 자산을 재사용하기 어렵고 새로운 화면 흐름(날짜 범위 선택, 달력 UI 결과 표시)이 필요해 "심층 분석처럼 확장 화면 하나 추가" 패턴과는 규모가 다르다.

목적별로 요구하는 입력값도 제각각이라는 점이 재확인됨 — 예: 信운세 이사택일은 생년월일시·성별 외에 **이사 방향(중앙·동·남동 등 8방향)**까지 추가로 입력받는다. 즉 "결혼 택일 화면"과 "이사 택일 화면"은 같은 택일이라도 입력 필드 구성 자체가 다르다 — 하나의 범용 "택일 화면"으로 묶기 어렵고 목적별 화면 설계가 필요.

## (갱신) 경쟁 앱들의 제품 형태 — 예상보다 다양함, "독립 상품"만은 아니다
기존 결론("핵심 정체성과 다른 별도 신규 기능")은 유효하지만, 실제 시장에서 택일이 어떤 형태로 제공되는지 구체 사례를 보강한 결과 제품 형태는 최소 두 갈래로 나뉜다.

### (A) 대형 종합 운세 앱이 내부 메뉴/기능으로 통합해서 제공
- **점신**(누적 다운로드 1,500만 이상, 업계 1위권): 이사택일 등을 앱 내부 운세 메뉴 중 하나로 제공 — 별도 결제 페이지가 아니라 종합 앱의 콘텐츠 카테고리 하나로 녹아 있음.
- **포스텔러**: "운세 캘린더"라는 전용 화면으로 "계약/미팅/이사/면접 등 중요한 날의 일정을 잡아보세요", "3000년 전통의 정밀한 택일 계산법으로 당신의 길일을 밝혀드려요"라는 카피로 소개. 결제는 앱 전체에서 쓰는 가상화폐("포스", 500포스 ₩7,700 ~ 7,200포스 ₩97,000)로 이뤄져 택일만을 위한 별도 결제 흐름은 아님 — 달력 UI 자체가 핵심 차별점.
- **사주풀이도우미(fateup.com)**: "≡ 택일/출산" 독립 메뉴가 있으나 하위 항목(약혼/결혼 택일, 이사 택일 등) 다수가 "준비 중" 상태 — 택일 만세력(무료)만 실제 제공, 나머지는 유료 예정.

### (B) 독립 랜딩페이지형 유료 단일 상품
- **信운세**: `/goodjoin`(결혼택일), `/goodmove`(이사택일) 등 목적별로 완전히 분리된 URL/페이지. 천기대요(天機大要) 택일법을 기본 채용한다고 명시. 결제 후 "핸드폰번호+비밀번호"로 일정 기간 결과 재확인 — 앱 계정 시스템과 무관하게 건별 결제.
- **사주하루**: 제왕절개 등 출산 택일 전용 페이지(`/taegil`) — 역시 목적 단일 상품.

**갱신된 결론**: "택일은 독립 신규 제품"이라는 기존 판단의 방향은 유지하되, 실제로는 대형 앱일수록 내부 메뉴(달력 UI)로 흡수하는 경향도 있다는 점이 확인됨. 다만 이 앱들도 택일 로직 자체는 기존 명식 해석 파이프라인과 별개의 "기간 스캔형" 계산이라는 점은 동일 — 화면을 어디에 붙이느냐(독립 상품 vs 메인 앱 내부 메뉴)의 제품 배치 문제일 뿐, 계산·입력 모델의 이질성이라는 핵심 결론은 바뀌지 않는다.
