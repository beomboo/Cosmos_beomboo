# 대운(大運)·세운(歲運) — 콘텐츠 갭

_최종 확인: 2026-07-19_

**현황**: 미구현. `BirthInfo.gender`의 doc-comment가 "대운 순행/역행 방향에 필요"라고 이미 밝히고 있으면서도 정작 대운 계산은 없는 상태. 구현 여부는 사람 결정 대기.

## 개념
- **대운(大運)**: 월주 기준 10년 주기로 바뀌는 운의 흐름. 8~10번의 대운이 인생 전체를 관통.
- **세운(歲運)**: 그 해의 간지(60갑자)로 정해지는 1년 단위 운. 설날이 아니라 **입춘 기준**으로 해가 바뀐다 — 연주(년주) 계산과 완전히 동일한 규칙(입춘 절입시각 이전 출생이면 전년도 간지 적용)이므로, 특정 연도의 세운을 구하는 것은 곧 "그 연도 입춘을 기준으로 한 연주 계산을 올해에 적용"하는 것과 같다.
- **방향(순행/역행)**: 남자는 년간이 양간(甲丙戊庚壬)이면 순행·음간(乙丁己辛癸)이면 역행, 여자는 정반대(양남음녀=순행, 음남양녀=역행) — `BirthInfo.gender`가 정확히 이 규칙 때문에 필요.
- **대운수(첫 교운 나이)**: 출생일로부터 다음(순행)/이전(역행) **절(節)**까지의 날짜·시간을 3으로 나눈 값. "절기 간격 약 30일 : 대운 10년 주기" 비례식(3일=1년)이 근거. 간이 계산에서는 나머지 처리 규칙(나머지 1은 버림, 나머지 2는 올림 — 반올림과 동일)을 쓰고, **24절기 전부가 아니라 절입(節入)에 해당하는 12개("절")만 사용**한다(입춘·경칩·청명·입하·망종·소서·입추·백로·한로·입동·대설·소한 — 짝수 인덱스). **대운수 계산 자체가 정밀한 절기 절입 시각을 요구** — manseryeok의 `luck-pillars.ts`가 `solarTermInstantMs()`(Meeus 공식 기반)를 직접 호출함을 코드로 확인했고, `saju`(Dart)의 `core/solar_terms.dart`도 동일하게 `sunApparentLongitude()`(태양 겉보기 황경, Meeus 계열)로 절 절입 시각을 실제 계산해 대운수 산출에 그대로 사용함을 코드로 재확인(2026-07-19, 아래 "오픈소스 참고 구현" 절). 세부 대운수는 "3일=1년, 1일=4개월, 1개월=30일" 환산으로 년·월·일까지 잔여 보존.
- **대운수 범위**: 절 간격이 최대 약 30일이므로 대운수는 대략 **1~10세** 범위(0으로 나오는 경우도 실무에서는 최소 1세로 보정하는 것이 일반적 — manseryeok 코드도 `Math.max(1, Math.round(days / 3))`로 하한을 1세로 강제). 다만 `saju`(Dart)는 이 하한을 강제하지 않아 이론상 0세가 나올 수 있다는 예외를 코드 분석으로 발견 — 아래 "오픈소스 참고 구현" 절 참고.

## 난이도 차이 — 대운과 세운은 완전히 다르다
- **대운은 [계산_정확도_이슈.md](계산_정확도_이슈.md)의 "정밀 절기 계산"에 직접 의존** — 지금 우리 절기 근사(입춘=2/4 고정)로는 대운수가 며칠~몇 주 단위로 부정확해질 수 있다. 콘텐츠만 추가하면 되는 문제가 아니라 계산 정확도 이슈와 강하게 엮여 있음.
- **세운은 이미 있는 로직을 그대로 재사용 가능** — "특정 연도의 년주"일 뿐이므로 `four_pillars.dart`의 년주 계산 로직(입춘 기준 연도 판정+60갑자 순환)을 "올해"에 적용하면 된다. 새 정밀도 문제 없이 지금 코드베이스만으로 구현 가능한 낮은 문턱의 확장. `saju`(Dart)의 세운 함수도 정밀 절기 계산 없이 단순 정수 연산만으로 구현돼 있어 이 판단을 코드 레벨로도 뒷받침한다(아래 참고).

**콘텐츠 확장 우선순위: 세운 > 십신 > 용신 > 대운** (구현 문턱이 낮은 순).

## 시장 노출 현황
"공명만세력"은 "대운·세운·월운 흐름 분석"을 핵심 기능으로 내세우고, "사주한눈에"는 "대운 타임라인"(10년 주기 시각화), "사주팔자(Sia Makerlab)"는 "일생·대운·세운·월운·일운" 드래그 차트 제공 — 전문/특화 만세력 앱에서는 표준 기능. 반면 포스텔러 캐주얼 메인 앱에는 없고 "포스텔러 만세력" 전문가용 서브 앱에만 있음(십신과 같은 패턴) — 캐주얼 앱에서 아예 다루지 않는 것도 흔한 선택지.

## 교차 검증 (2026-07-16)
아래 3개 이상의 상호 독립적 출처 + 실제 오픈소스 코드로 순행/역행 판단·대운수 계산·세운 기준을 재검증했다.

- **순행/역행 규칙**: 위키백과, KNS뉴스통신 기사, threads/brunch 게시물, manseryeok 코드 주석 4곳 모두 "양간 남자·음간 여자=순행, 음간 남자·양간 여자=역행"으로 일치. 기존 문서 서술과 일치 — 수정 불필요.
- **대운수 계산(절÷3)과 나머지 처리**: 위 4개 출처 모두 "다음(순행)/이전(역행) 절까지 일수 ÷ 3, 나머지 1은 버림·나머지 2는 반올림"으로 일치. manseryeok 실제 코드(`Math.round(days / 3)`)도 동일한 반올림 방식으로 구현되어 있어 코드 수준까지 일치를 확인했다.
- **대운수 범위 오류 발견 및 수정**: 기존 문서는 "대략 2~20세"로 서술돼 있었으나, 절(12개, 약 30일 간격) 기준 최대 일수가 약 30일이므로 30÷3=10이 이론적 상한이고, 역학동·brunch 등 실사례("3세, 6세, 8세, 10세가 넘어서야 첫 대운" 등)도 모두 1~10 범위 안에 있었다. manseryeok 코드의 `Math.max(1, Math.round(days / 3))`도 하한 1세·구조적 상한 10세를 명시적으로 보여준다. **"2~20세"는 오기로 판단해 "1~10세"로 정정했다.**
- **세운=입춘 기준**: 나무위키 "입춘" 문서, 에이퓨어 블로그 모두 "설날이 아니라 입춘 절입시각을 기준으로 연간지가 바뀐다"고 명시 — 기존 문서의 "특정 연도의 년주"라는 서술과 일치, `기초개념.md`의 "입춘 기준 연도 판정" 서술과도 모순 없음. 수정 불필요.
- **기초개념.md·계산_정확도_이슈.md와의 정합성**: 두 문서 모두 "대운=성별+정밀 절기 절입 시각 필요(★★★★☆)", "세운=조회 연도만 필요, 절기 정밀 계산 불필요(★☆☆☆☆)"로 서술 — 이번 문서의 난이도 차이 서술과 완전히 일치. 추가 불일치 없음.

## 오픈소스 참고 구현 — saju(Dart) `core/luck.dart` 코드 분석(2026-07-19)

[오픈소스_라이브러리.md](오픈소스_라이브러리.md)가 유력 채택 후보로 꼽은 `saju`(Dart, `gracefullight/pkgs`) 패키지의 `core/luck.dart`(대운·세운·월운·일운)와, 실제로 그 함수를 호출하는 `saju.dart`(공개 API), 그리고 절기 절입 시각을 계산하는 `core/solar_terms.dart`를 함께 읽어 대조했다(용신·십신 코드 분석 때와 동일한 방식, GitHub main 브랜치 소스 직접 확인).

### 1. 순행/역행 판단 — 문서 서술과 완전히 일치
```dart
final isForward =
    (yearStemPolarity == Polarity.yang && gender == Gender.male) ||
    (yearStemPolarity == Polarity.yin && gender == Gender.female);
```
"양간 남자·음간 여자=순행, 그 외=역행"을 그대로 구현 — 문서 서술과 일치, 수정 불필요.

### 2. 정밀 절기 절입 시각을 실제로 쓴다 — "가장 중요한 확인 지점" 확인 완료
`calculateMajorLuck()` 자체는 `nextJieMillis`/`prevJieMillis`를 매개변수로 받기만 하고 직접 계산하지 않는다. 그래서 이 값이 어디서 오는지가 핵심인데, 실제 공개 API `saju.dart`의 `getSaju()`를 보면:
```dart
final solarTerms = analyzeSolarTerms(dtLocal); // core/solar_terms.dart
final majorLuck = calculateMajorLuck(
  ...,
  nextJieMillis: solarTerms.nextJieMillis,
  prevJieMillis: solarTerms.prevJieMillis,
);
```
그리고 `analyzeSolarTerms()`는 `sunApparentLongitude(dtUtc)`(`utils/solar_longitude.dart`, 태양 겉보기 황경 계산, 만세력에서 이미 확인한 Meeus 계열 공식과 계수까지 일치)로 24절기 인덱스를 구해 실제 절입 날짜를 이진 탐색(`findTermUtc`)으로 찾아낸다. 즉 **이 패키지의 대운수 계산은 실제 운영 경로에서 항상 정밀 태양 황경 기반 절입 시각을 쓴다** — "절기를 그레고리력 날짜로 고정하는 근사"는 전혀 쓰지 않음. 이는 manseryeok의 `solarTermInstantMs()` 확인 결과와 정확히 같은 결론이라, "대운수는 정밀 절기 계산에 의존한다"는 문서의 기존 서술을 코드 레벨로 한 번 더 보강한다. (참고: `calculateMajorLuck()`에 절기 정보를 안 넘기면 `daysToTerm = 30`으로 대체하는 방어적 기본값이 있지만, 실제 파이프라인에서는 항상 정밀값이 주입되므로 이 기본값이 쓰일 일은 없다.)

또한 절입 시각을 찾는 `_findPrevJie`/`_findNextJie` 함수는 `jieIndex`가 항상 짝수(even index)가 되도록 강제한다(24절기 중 짝수 인덱스만 "절"에 해당) — "24절기 전부가 아니라 절입 12개만 쓴다"는 문서 서술과 일치.

### 3. 대운수(나머지 처리) — 결과는 같지만 계산 경로는 다르다, 하한 미보정은 새 발견
```dart
final totalMonths = ((daysToTerm / 3) * 12).round();
final years = totalMonths ~/ 12;
final months = totalMonths % 12;
final startAge = months >= 6 ? years + 1 : years;
```
이 코드는 문서가 서술한 "일수÷3의 나머지가 1이면 버림·2면 올림"을 문자 그대로 구현하지 않는다. 대신 "일수÷3을 개월수(×12)로 환산해 반올림 → 그 개월수를 다시 년/월로 분해 → 개월이 6개월 이상이면 1년 올림"이라는 별도 경로를 쓴다. 여러 일수 값(25/26/27/28/29일)으로 직접 계산해본 결과 두 방식은 **동일한 나이를 반환**함을 확인했다(예: 26일→9세, 29일→10세로 manseryeok의 `Math.round(days/3)`과 일치) — 계산 경로는 다르지만 결과는 문서가 정리한 규칙과 일치한다. **다만 문서가 함께 서술한 "하한 1세 강제(`Math.max(1, ...)`)에 대응하는 로직"은 이 코드에 없다.** `startAge` 계산에 하한 보정 코드가 전혀 없어, 절입 시각으로부터 약 1.25일 이내에 태어난 경우 이론상 대운수가 0으로 나올 수 있다(`totalMonths`가 0~5일 때 `years=0, months<6`이라 `startAge=0`). manseryeok은 `Math.max(1, ...)`로 이 경우를 명시적으로 1세로 올림 처리하는데, `saju`(Dart)는 이 보정이 빠져 있다 — 실제 채택 시 이 하한 처리를 별도로 추가해야 함을 새로 발견했다. 문서의 "1~10세" 범위 서술 자체(대다수 출처가 뒷받침하는 표준)는 유효하게 유지하되, 이 특정 패키지는 하한을 지키지 않는 예외로 별도 기록한다.

### 4. 세운(歲運) — 별도 함수로 존재, 절기 계산과 무관
`calculateYearlyLuck(birthYear, fromYear, toYear)`와 `getYearPillar(year)`가 세운 전담 함수다. 로직은 매우 단순하다.
```dart
final idx60 = (((year - 1984) % 60) + 60) % 60; // 1984 = 갑자년(기준 연도)
final pillar = Pillar.fromIndex(idx60);
final age = year - birthYear + 1; // 세는나이(한국식) 기준
```
1984년을 갑자년(60갑자 순환의 기준점, index 0)으로 고정하고 정수 나눗셈만으로 해당 연도의 간지를 구한다 — 절기 절입 시각 계산이 전혀 필요 없다. 이는 "세운은 특정 연도의 년주일 뿐이라 정밀 절기 계산과 무관하다"는 문서의 난이도 차이 서술을 코드 레벨로 뒷받침한다. 다만 이 함수는 입력이 이미 "몇 년도인지" 정수 하나뿐이라 입춘 절입일 자체를 다시 판정하지 않는다 — 즉 "그 해 전체를 통틀어 하나의 간지로 본다"는 통상적 세운 표기 관행(대부분의 만세력·운세 콘텐츠가 채택하는 방식)을 그대로 따르는 것이지, 연초(1/1~입춘 전) 세운을 전년도 간지로 세분화하는 방식은 아니다. 우리 문서의 "특정 연도의 년주 계산을 올해에 적용" 서술과 부합하며 별도 모순은 없다.

부수적으로 `calculateMonthlyLuck()`(월운, 오호둔 규칙 `baseMonthStemIdx = (yearStemIdx*2+2)%10` 기반)과 `calculateDailyLuck()`(일운, 우리가 이미 [오픈소스_라이브러리.md](오픈소스_라이브러리.md)에서 확인한 것과 같은 JDN 기반 일주 계산 재사용)도 같은 파일에 구현돼 있다 — 이번 조사 범위(대운·세운)는 아니지만 향후 월운/일운 콘텐츠 조사 시 참고할 것.

### 결론
- 순행/역행 규칙, 절 12개만 사용하는 규칙, 대운수의 정밀 절기 의존성 → **코드 레벨로도 확인됨**(문서 서술과 일치, 근거 보강).
- 대운수의 나머지 처리 결과값 → **일치**(계산 경로만 다름, 최종 나이는 같음).
- **대운수 하한 1세 강제** → `saju`(Dart)에는 없음(신규 발견, 채택 시 보정 필요).
- 세운 계산 → 별도 함수(`calculateYearlyLuck`/`getYearPillar`)로 존재하며 정밀 절기 계산이 필요 없다는 서술과 일치.
