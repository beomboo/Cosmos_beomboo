# 십신(十神) — 콘텐츠 갭

_최종 확인: 2026-07-19_

**현황**: `project/lib/` 전체에 십신 관련 로직(비견·겁재·식신·상관·편재·정재·편관·정관·편인·정인)이 전혀 없음. 오행 분포와 함께 명리학의 양대 해석 축인데 우리 앱은 오행만 다룬다. 구현 여부는 사람 결정 대기 — 아래는 실제 구현 시 참고할 콘텐츠·계산 근거.

## 계산 방식
일간(day master)의 오행·음양과 대상 글자(천간/지지)의 오행·음양을 비교하는 5단계 분기(같은 오행 / 일간이 생함 / 일간이 극함 / 대상이 일간을 극함 / 대상이 일간을 생함)로 비견~정인 10개를 산출 — 절기·정밀도와 완전히 무관한 단순 룩업 로직. 지지의 십신은 지장간 전체가 아니라 **지장간 본기(그 지지를 대표하는 천간 하나)만** 사용하는 것이 표준(가장 단순한 쪽 채택) — 『명리약언』도 지장간 중 본기(정기)를 중심으로 보라고 명시하는 것으로 확인됨(웹 검색 교차 확인, 2026-07-16) — 이후 saju(Dart) `core/ten_gods.dart` 코드 분석으로도 동일하게 확인됨(2026-07-19, 아래 "오픈소스 참고 구현" 절 참고).

이미 `FourPillars`가 각 기둥의 stemIndex/branchIndex를 갖고 있어, 순수 계산 로직(테이블 룩업 수준)만 추가하면 된다 — 새 외부 라이브러리·AI 연동 불필요. 심층 분석(MBTI·관심사)과 비슷한 구현 난이도.

## 십신 도출 규칙표 — 검증 완료(2026-07-16)

일간 기준 대상 글자와의 "오행 관계"(같음/일간이 생함/일간이 극함/일간을 극함/일간을 생함) × "음양 일치 여부"(같음/다름) 조합으로 10개가 정해진다.

| 오행 관계 (일간 → 대상) | 음양 같음 (偏 계열) | 음양 다름 (正 계열) |
|---|---|---|
| 오행이 같음 | 비견(比肩) | 겁재(劫財) |
| 일간이 생(生)함 | 식신(食神) | 상관(傷官) |
| 일간이 극(剋)함 | 편재(偏財) | 정재(正財) |
| 대상이 일간을 극함 | 편관(偏官, 칠살) | 정관(正官) |
| 대상이 일간을 생함 | 편인(偏印, 효신) | 정인(正印, 인수) |

핵심 원리: **음양이 같으면(양-양, 음-음) "편(偏)/비(比)/겁(劫)" 계열, 음양이 다르면(양-음, 음-양) "정(正)" 계열**로 갈린다 — 비견·겁재 쌍만 이름에 偏/正이 직접 안 붙을 뿐 같은 음양 규칙을 따른다.

**아래 5개의 상호 독립적인 출처를 대조해 완전히 일치함을 확인했다** — 오탈자·불일치 없음(직전 사이클 십이운성 검증과 동일한 방식).
1. manseryeok(TypeScript) 소스 `src/features/ten-gods.ts` (`getTenGod` 함수) — 코드 레벨 대조. 실제 raw 파일을 직접 받아 확인(2026-07-16): `sameYinYang`이 true면 비견/식신/편재/편관/편인, false면 겁재/상관/정재/정관/정인을 반환하는 로직이 위 표와 정확히 일치.
2. [위키백과 "십성 (사주팔자)"](https://ko.wikipedia.org/wiki/%EC%8B%AD%EC%84%B1_(%EC%82%AC%EC%A3%BC%ED%8C%94%EC%9E%90)) — "일간과 오행·음양이 같으면 비견, 오행은 같지만 음양이 다르면 겁재..." 서술이 표와 일치.
3. [사주 초급 - 십성(十星)·십신(十神)·육친(六親) 정리 - 사주스터디](https://www.sajustudy.com/53) — 오행 관계×음양 관계 표를 별도로 제공, 10칸 전부 일치.
4. [10강 십성의 의미 - 정해 만세력](https://guide.8-codes.com/lecture/elementary/10.html) — "음양 같음=편(강렬), 음양 다름=정(유순)" 원칙 서술이 일치. (예시: 정재=고정 수입, 편재=횡재로 편/정 성격 차이도 부합)
5. `saju`(Dart) 패키지 `core/ten_gods.dart`(`getTenGodKey` 함수) — GitHub `gracefullight/pkgs` main 브랜치 raw 코드 직접 대조(2026-07-19). 5단계 분기·반환값이 위 표와 1:1로 일치, manseryeok에 이어 두 번째 독립 코드베이스 검증.

다섯 출처가 서로 다른 형태(TS 코드, Dart 코드, 백과사전, 강의 사이트, 표)임에도 규칙이 완전히 일치하므로 잔여 리스크는 낮다고 판단한다.

## 오픈소스 참고 구현 — saju(Dart) `core/ten_gods.dart` 코드 분석(2026-07-19)

`오픈소스_라이브러리.md`가 유력 채택 후보로 꼽은 `saju`(Dart) 패키지의 실제 소스(GitHub `gracefullight/pkgs` main 브랜치, `packages/saju-dart/lib/src/core/ten_gods.dart`·`types/hidden_stems.dart`)를 직접 읽어 대조했다(용신 코드 분석 때와 동일한 방식).

**1) 천간 십신 도출 로직 — 규칙표와 완전히 일치**

`getTenGodKey(dayMaster, targetStem)` 함수는 위 규칙표와 정확히 같은 5단계 분기를 그대로 구현한다(순서·조건·반환값 전부 일치):
- 같은 오행: `samePolarity` ? 비견(companion) : 겁재(robWealth)
- 일간이 생함(`dmElement.generates == targetElement`): `samePolarity` ? 식신(eatingGod) : 상관(hurtingOfficer)
- 일간이 극함(`dmElement.controls == targetElement`): `samePolarity` ? 편재(indirectWealth) : 정재(directWealth)
- 대상이 일간을 극함(`targetElement.controls == dmElement`): `samePolarity` ? 편관·칠살(sevenKillings) : 정관(directOfficer)
- 대상이 일간을 생함(`targetElement.generates == dmElement`): `samePolarity` ? 편인(indirectSeal) : 정인(directSeal)

manseryeok에 이어 두 번째 독립 코드베이스로 코드 레벨 검증이 추가된 셈이다.

**2) 지지(branch)의 십신 — "본기만 사용" 서술이 코드 레벨로 확인됨, 다만 구조는 이 문서 기존 서술보다 조금 더 미묘함**

이번 조사의 핵심 확인 지점. `HiddenStems.forBranch(branch)`는 지장간 전부(1~3개, 본기/중기/여기)를 weight와 함께 반환하고, `_analyzePillar()`는 실제로 지장간 전부에 대해 십신을 계산해 `BranchTenGod.hiddenStems` 리스트에 저장한다 — 즉 지장간 여러 개의 십신을 코드가 다 버리는 건 아니다.

그러나 "지지 하나를 대표하는 단일 십신 값"(`BranchTenGod.tenGod`, `countTenGods()` 집계 함수가 실제로 사용하는 값)은 `hiddenStemTenGods.first.tenGod` — 즉 리스트의 첫 원소만 쓴다. `HiddenStems.data` 맵을 확인하면 각 지지의 지장간 리스트는 항상 `본기(primary) → 중기(secondary) → 여기(tertiary)` 순서로 정렬돼 있고, `HiddenStems.primaryStem()`도 `stems.first.stem`을 반환하므로 `.first` == 본기와 동일하다. 별도 편의 함수 `getTenGodForBranch()`의 독스트링도 "Each branch contains one or more hidden stems... uses the primary (본기) hidden stem"이라 명시하며 `HiddenStems.primaryStem(branch)`를 직접 호출한다.

**결론**: "지지 십신 집계는 본기만 사용"이라는 이 문서의 기존 서술(2026-07-16, 웹 검색 근거)은 코드 레벨로도 확인됨 — 정정 사항 없음. 다만 정확히는 이 코드가 "본기 외 지장간의 십신도 계산은 해두고(중기·여기까지 리스트로 노출), 대표/집계용 단일 값에서만 본기를 쓰는" 구조라는 점이 이번에 새로 확인된 세부 사항이다. 우리가 향후 구현할 때도 "지지당 대표 십신 1개(본기 기준)"를 기본으로 하되, 필요하면 중기·여기 십신을 보조 정보로 함께 노출하는 확장이 자연스럽다는 참고가 된다.

**3) 참고: 지장간 표 자체의 사소한 차이 (십신 로직과는 무관)**
이 코드의 `Branch.wu`(午) 지장간은 정(丁, 0.7)·기(己, 0.3) 2개로만 구성돼 있다(일부 전통 조견표는 병(丙)을 초기로 추가해 3개로 보기도 함). 이는 십신 도출 로직의 문제가 아니라 지장간 조견표 자체의 유파 차이이므로 이 문서 범위를 벗어난다(지장간 조견표 자체를 다루는 별도 개념 문서가 생기면 그쪽에서 검증할 사항).

## 콘텐츠 설계 — "5그룹 단순화 + 실생활 문장" 실제 사례

십신 10개를 그대로 노출하지 않고, 음양 짝을 묶어 5그룹으로 단순화 + 한 줄 실생활 문장을 붙이는 방식이 실제로 쓰인다:

| 그룹 | 묶이는 십신 | 한 줄 의미 | 실생활 예시 |
|---|---|---|---|
| 비겁(比劫) | 비견·겁재 | 나 자신, 자존심, 독립성 | "나는 내 방식대로 할래" |
| 식상(食傷) | 식신·상관 | 표현, 아이디어, 창작 | "내가 생각한 아이디어가 있는데..." |
| 재성(財星) | 정재·편재 | 돈, 물질, 현실 감각 | 정재=월급형 안정 수입, 편재=투자·사업 소득 |
| 관성(官星) | 정관·편관 | 직장, 책임감, 조직 생활 | 정관=대기업/공무원, 편관=스타트업/계약직 |
| 인성(印星) | 정인·편인 | 학문, 배움, 사고방식 | 정인=정규 교육, 편인=독학 |

이 방식은 "정관 vs 편관" 등 헷갈리는 용어 비교표까지 함께 제공해 초심자가 세부 10개를 몰라도 5그룹 수준에서 이해할 수 있게 한다 — "전문 용어를 일상어로 재해석"이라는 성공 공식과 일치([../운세/UIUX_트렌드.md](../운세/UIUX_트렌드.md) 참고). 이 5그룹 명칭(비겁·식상·재성·관성·인성)도 위 5개 출처 전부에서 동일하게 확인됨.

"명리.kr" 서비스는 ①만세력 변환 → ②오행 분포+일간 기준 십신 도출 → ③AI 종합 풀이, 3단계 구조를 쓴다. 우리 앱(오행 분포 → 규칙 기반 카테고리 풀이) 다음 단계에 "우세 십신 그룹" 판정을 끼우면 유사하게 확장 가능.

## 노출 방식 — 경쟁 앱은 "완전 분리형" 정보구조를 씀

포스텔러는 캐주얼 메인 앱과 별도로 **"포스텔러 만세력"이라는 전문가용 서브 앱**을 두고, 거기서만 십신을 "열 개의 별 차트"+용어 해설로 노출한다. 점신·헬로우봇은 반대로 **단일 앱 안에서 점진적으로 심화 콘텐츠를 노출**하는 방식(3강 중 2곳이 이쪽). 신운세는 애초에 캐주얼 레이어가 없는 전문가 전용 도구.

**시사점**: 우리는 캐주얼 메인 앱 하나만 있으므로, 포스텔러식 완전 분리보다는 **단일 앱 점진 노출형**이 규모·구조에 맞다 — 이미 있는 "심층 분석(MBTI·관심사)" 확장 화면 패턴을 그대로 십신에도 적용 가능(메인 결과 화면은 오행 중심으로 가볍게, 십신은 별도 확장 화면으로).
