# 오픈소스 사주 계산 라이브러리

_최종 확인: 2026-07-18_

계산 엔진·참고 구현 후보 비교. 실제 채택 여부는 전부 사람 결정 대기(`PROJECT_ROUTER.md` "사주 계산 로직" 행 참고) — 이 문서는 결정 시 바로 참고할 수 있도록 실측 정보를 정리해둔다.

## 0. saju (Dart) — 유일한 Dart 네이티브 후보, 가장 유력한 채택 후보

- **pub.dev**: https://pub.dev/packages/saju (게시 버전 0.1.1) / **GitHub**: https://github.com/gracefullight/pkgs/tree/main/packages/saju-dart (main 브랜치 0.1.3 — pub.dev보다 앞서 있음)
- **우리 기술 스택(Flutter/Dart)과 완전히 같은 언어** — `flutter pub add saju`로 바로 의존성 추가 가능. 나머지 후보는 전부 TS/Python이라 포팅 필요.
- MIT 라이선스, 검증된 퍼블리셔(gracefullight.dev). 다만 아직 초기 단계(pub.dev 기준 다운로드 32회/30일, 좋아요 4, pub points 140) — 프로덕션 신뢰도·유지보수 지속성 미검증.
- **재확인(2026-07-18)**: pub.dev 게시 버전(0.1.1)·GitHub main 브랜치(0.1.3) 모두 지난 확인(2026-07-15) 이후 변동 없음. GitHub 커밋 로그상 `packages/saju-dart` 경로 마지막 커밋은 2026-02-21(`chore: release main`, 0.1.3 릴리스)이 여전히 최신 — 약 5개월째 업데이트가 멈춰 있다. `sinsals.dart`(신살 41종)도 여전히 pub.dev에는 미게시(0.1.1에는 없고 GitHub main 0.1.3에만 존재). 즉 "초기 단계·유지보수 정체" 우려가 이번 재확인으로 오히려 강화됐다 — 채택 전 저자에게 pub.dev 재배포 계획을 문의하거나, 포크해서 직접 관리하는 옵션도 고려할 것.
- 의존성: `jiffy`(날짜)·`timezone`(IANA tzdata)·`lunar`(음력 변환, 다른 pub.dev 패키지). 테스트 3개 파일(`four_pillars_test.dart`·`saju_test.dart`·`ten_gods_test.dart`) 존재.

**일주 계산이 율리우스일 수(JDN) 기반**(`dayPillarFromDate`, `(jdn-11)%60`)이라 `DateTime.difference()` 같은 로컬 타임존 산술을 아예 안 씀 — 우리가 겪은 "서머타임 기간 DateTime.difference() 하루 밀림 버그"가 이 패키지 설계에서는 애초에 발생 불가능한 구조.

참고로 GitHub main에는 `sinsals.dart`·`solar_terms.dart` 외에 `core/nayin.dart`(납음오행)도 존재한다(0.1.3 체인지로그의 "add sinsals, nayin, solar term monthly luck" 항목) — 납음오행은 우리 사람 결정 대기 항목에는 아직 없는 개념이라 이 표에는 포함하지 않았지만, 신살·십이운성처럼 향후 조사 대상이 될 수 있다.

### 검증 상태
십이운성·신살 조견표는 웹 검색 + 이 패키지 코드(`_yangStemBirthBranch` 등, `sinsals.dart`의 11개 맵) 양쪽으로 대조해 완전히 일치함을 확인했다. **다만 "우리 `four_pillars_test.dart`의 기존 케이스와 실제 수치가 일치하는가"는 아직 미검증** — 코드 리딩만으로는 확인됐지만 실제 설치 후 나란히 실행하는 교차 검증은 남아 있다(새 의존성 추가는 채택 여부와 얽혀 있어 신중 필요).

## 1. manseryeok (TypeScript) — 가장 많이 참고한 계산 엔진
- https://github.com/yhj1024/manseryeok, MIT
- KASI 정본 절기 데이터 기반, 1800~2300년(양력) 지원. 런타임 의존성 없음, ~40KB 경량.
- **입력**: 연/월/일/시(24h)/분, 양력·음력, 진태양시 보정, 자시 처리 방식, 성별(대운용) / **출력**: 4기둥, 오행/음양, 십신, 공망, 대운
- Dart 네이티브가 아니라 포팅이 필요하지만, 알고리즘 구조 자체는 `saju`(Dart)와 같은 계열(Meeus 공식)이라 상호 검증 자료로 유용. 상세 알고리즘은 [계산_정확도_이슈.md](계산_정확도_이슈.md) 참고.

## 7. bazi-skill — 다중 유파 AI 오케스트레이션 harness (Dr.Saju 아키텍처의 발전형, 2026-07-15 발견)
- https://github.com/xuemian168/bazi-skill — V2EX 인디 개발자 커뮤니티에서 발견(상세 커뮤니티 반응·UX 피드백은 [../운세/SNS_콘텐츠_전략.md](../운세/SNS_콘텐츠_전략.md)의 "SNS 채널 재시도 결과" 절 참고).
- "AI 사주 앱"이 아니라 **명리 전용 AI 오케스트레이션 harness**를 표방 — 5번 Dr.Saju와 같은 "계산은 결정론, 해석은 AI" 원칙을 한 단계 더 세분화한 구조.
  - **결정론 계층(코드)**: 사주 배치, 진태양시 보정, 시주(時柱) 경계가 모호한 후보 시각 처리, 결과 JSON 스키마 검증까지 전부 코드가 담당 — AI는 이미 확정된 사실만 놓고 해석한다(AI가 사주 자체를 "지어내는" 것을 구조적으로 차단).
  - **다중 유파 합의 계층(AI)**: 자평(子平)·왕쇠(旺衰)·조후(調候)·맹파(盲派)·자미두수(紫微斗數)·서양 점성술·택일까지 여러 유파를 별도 "역할(role)"로 분리해 각자 의견을 내게 하고, 마지막으로 "주리 플래너/주리관" 역할이 유파 간 이견을 조정해 하나의 구조화된 리포트로 합성.
- 우리가 아직 답을 정하지 못한 "유파 간 해석이 서로 다를 때 사용자에게 어떻게 보여줄지" 문제에 대한 구체적인 설계 참고 사례 — 다만 **AI 해석 연동은 `CLAUDE.md` 규칙 6번의 게이트 항목**이라 실제 채택 여부는 여전히 사람 결정 대기, 이 항목은 아키텍처 참고 자료로만 기록한다.
- **라이선스 확인 완료(2026-07-18)**: 저장소에 `LICENSE` 파일이 없고 README·SKILL.md 어디에도 라이선스 고지가 없다 — GitHub 기본 원칙상 라이선스 미명시 저장소는 저작권자가 모든 권리를 보유한 상태(사실상 all rights reserved)로 간주해야 한다. 코드나 조견표를 그대로 옮겨 적는 방식의 참고는 저작권 문제가 될 수 있으므로, "유파 간 이견 조정" 같은 **아키텍처 아이디어 참고 수준**으로만 활용하고 실제 텍스트·프롬프트·조견표를 복제하지 않을 것.

## 8. baziflow-core (TypeScript, 신규 발견·2026-07-18)
- https://github.com/Eastern-Sunrise/bazi-core (npm: `baziflow-core`) — 생성일 2026-07-11, MIT, 현재 1 star. **매우 최근에 생성된 저장소라 검증 데이터가 거의 없음**(신뢰도 판단 보류, 참고용으로만 기록).
- 사주팔자(연/월/일/시)·진태양시·오행 분포·십신·장간(藏干)·납음(納音)·대운까지 계산하되, README에 "This package computes the **chart**, not a reading... no favourable/unfavourable elements, no strength verdict, no interpretation"이라고 명시 — 계산과 해석을 의도적으로 분리한 설계 철학이 5번 Dr.Saju·7번 bazi-skill의 "계산은 결정론, 해석은 별도" 원칙과 정확히 같은 방향.

## 채택 방향 제안 (사람 결정 대기)
`saju`(Dart)가 우리 사람 결정 대기 항목 대부분(절기·자시·AST·십신·십이운성·용신·대운/세운)을 이미 구현했고 기술 스택도 일치해 가장 유력한 후보다. 다만 초기 단계 패키지라 신뢰도 검증(코드 품질 직접 검토·라이선스 재확인·기존 테스트와 교차 검증)이 우선 필요 — 특히 2026-07-18 재확인 결과 GitHub 커밋이 5개월째 정체돼 있어, 유지보수 지속성에 대한 우려를 채택 논의 시 반드시 함께 고려할 것. 설령 의존성으로 직접 쓰지 않더라도, 조견표·계산 로직은 이미 두 차례(웹 검색, 코드) 교차 검증됐으므로 상수만 옮겨 적는 "참고 구현"으로는 신뢰도가 높다.

각 저장소 라이선스는 실제 코드 반영 전 반드시 재확인할 것(`saju`(Dart)는 MIT 확인 완료, manseryeok·manseryeok-js도 MIT, baziflow-core도 MIT 확인 완료, **bazi-skill은 라이선스 파일 없음(사실상 all rights reserved) 확인 완료 — 아이디어 참고만, 텍스트/코드 복제 금지**).
