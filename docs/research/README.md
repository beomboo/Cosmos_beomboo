## 핵심 요약

**궁합(宮合) 첫 심층 조사 — 택일과 콘텐츠 갭의 "중간" 성격 확인(2026-07-08)**: 지금까지 결제 단가 사례("궁합 프리미엄 22,000원")로만 언급됐던 궁합을 처음으로 깊이 조사 — 겉궁합(띠만 비교, 시간 불필요)/속궁합(8자 전체 비교) 구분을 확인했고, 계산 구성요소를 뜯어보니 **오행 보완**(이미 있는 `ohaengCount` 재사용만으로 가능)·**천간합**(십신·십이운성과 동급 난이도의 고정 로직)까지는 쉽지만, **일지 합충형파**는 우리 코드에 전혀 없는 새 개념(어제 발견한 `saju`(Dart) 패키지의 `relations.dart`가 정확히 이 부분을 담당 — 참고 구현으로 연결됨)임을 확인. 입력 모델도 택일(완전히 다른 제품)만큼 이질적이지 않고 "1인분→2인분"으로 늘어나는 정도라, 콘텐츠 갭 6가지와 택일의 중간 성격으로 분류. 실제 시장에서 궁합은 고가 유료 콘텐츠로 취급되는 경우가 많아 "결제 로직" 사람 결정과 자연스럽게 얽히는 기능으로 보임. 자세한 내용은 [01_사주팔자_기초개념](01_사주팔자_기초개념/기초개념_정리.md)의 "궁합(宮合)" 절 참고. (구현 여부는 여전히 사람 결정 대기. 아직 존재하지 않는 2인 입력 화면에 대한 아이디어라 목업 파일은 수정하지 않음) **2026-07-08 같은 세션 후속 조사로 합충형파해 조견표 원문까지 확보 완료**: `relations.dart`에서 천간합(화하는 오행 포함)·육합·삼합·방합·충·해·파·형 전체 원문을 확보했고, 부수적으로 어제 미해결로 남겼던 "육해살 계산법"도 이 표의 "해(害)" 관계로 해결됨을 확인. 단, **`relations.dart`는 한 사람의 사주 내부 관계만 보는 함수(`analyzeRelations(FourPillars)`)라 궁합(2인 비교)에는 조견표만 재사용 가능하고 비교 로직 자체는 새로 작성해야 함**을 정확화(지난 사이클에 "그대로 참고 구현이 될 수 있다"고 적었던 것을 정정). 자세한 내용은 [01_사주팔자_기초개념](01_사주팔자_기초개념/기초개념_정리.md)의 "2026-07-08 후속 조사 — 합충형파해 조견표" 절 참고. **2026-07-08 같은 세션 마지막 후속 조사(UX·MVP)**: 실제 앱들은 사용자를 무제한 추가하는 다중 프로필 저장 방식을 쓰고(우리 `BirthInfoStore`는 1명 전용이라 확장 시 저장소 구조 자체를 바꿔야 함), 결과를 "100점+5단계 등급"으로 단순화해 보여주는 게 관례임을 확인. 겉궁합은 독립 MVP 화면이 아니라 "오늘의 띠별 궁합" 같은 일별 반복 콘텐츠로 존재하는 경우가 많아 "오늘의 운세류"에 더 가깝다고 재분류.

**⭐ 세션 최대 발견 — Dart 네이티브 사주 패키지 "saju" (2026-07-08)**: GitHub 코드 검색으로 우리와 완전히 같은 기술 스택(Dart/Flutter)으로 작성돼 pub.dev에 게시된 사주 라이브러리(`gracefullight/pkgs`의 `packages/saju-dart`, MIT 라이선스)를 발견. 사주팔자·십신·**십이운성**·**신살(41종)**·용신·신강신약·대운/세운/월운/일운·**정밀 절기**·**음력변환**·합충형파해까지, 지금까지 조사해온 "사람 결정 대기" 계산 항목 대부분을 이미 구현해뒀다 — 지난 두 사이클에서 각각 웹 검색만으로 도출한 십이운성·12신살 조견표를 이 패키지의 실제 코드와 대조해 **완전히 일치함을 확인**해 신뢰도까지 검증했다. 다만 아직 초기 단계(pub.dev 다운로드 34회, 버전 0.1.x)라 프로덕션 채택 여부는 별도 검증 필요. 자세한 내용은 [07_오픈소스_계산로직](07_오픈소스_계산로직/오픈소스_라이브러리.md)의 "Dart 네이티브 패키지 saju 발견" 절 참고. (채택 여부는 여전히 사람 결정 대기) **2026-07-08 같은 세션 후속 검증**: git 태그 대조로 **십이운성(`twelve_stages.dart`)은 pub.dev 게시 버전(0.1.1)에 이미 있어 지금 바로 `pub add` 가능하지만, 신살(`sinsals.dart`)은 아직 GitHub에만 있고 pub.dev엔 미게시**임을 확인. 절기 계산은 manseryeok과 동일한 Meeus 공식(계수 숫자까지 일치)임을 확인했고, **자시 관법(자정/23시 기준 선택)과 진태양시 보정을 이미 토글 가능한 옵션(`PillarPreset`)으로 구현**해뒀으며, 일주 계산이 율리우스일 수(JDN) 기반이라 우리가 이번 세션에 직접 겪은 "서머타임 기간 DateTime.difference() 하루 밀림" 버그가 이 패키지 설계에서는 애초에 발생할 수 없는 구조임도 확인 — 절기·자시·AST 세 가지 정확도 이슈가 이 패키지 하나로 동시에 해결될 잠재력이 구체적으로 확인됨. 자세한 내용은 [07_오픈소스_계산로직](07_오픈소스_계산로직/오픈소스_라이브러리.md)의 "saju(Dart) 패키지 후속 검증" 절 참고.

**⚠️ saju(Dart) 패키지에 대한 중요한 수정 — 음력 변환 의존성이 중국 전용 라이브러리(2026-07-08)**: `saju`의 `lunar: ^1.3.10` 의존성(6tail/lunar-flutter, pub.dev 좋아요 54·다운로드 7,730로 saju 본체보다 훨씬 성숙)이 저자 스스로 "Chinese Lunar" 전용이라고 명시한 라이브러리임을 확인. 한국(UTC+9)·중국(UTC+8)의 1시간 표준시 차이로 음력 합삭(초하루) 판정이 갈리는 경우가 1914~2099년 중 15번 있었다는 것도 확인(구정·추석 기준). **2026-07-08 같은 세션 후속 조사로 추측이 아니라 확정**: `LunarYear.dart`의 절기 계산 코드 주석에 개발자 본인이 "北京时间"(베이징 시간)이라고 명시해뒀음을 발견 — 절기·초하루 판정이 베이징 시간(UTC+8) 고정 기준으로 계산됨이 소스 코드로 확인됐다. **saju 패키지의 음력 변환 기능은 KST 보정 없이는 신뢰할 수 없음이 확정**, 절기·자시·AST·십이운성·신살 등 나머지 기능과는 무관한 별개의 위험. 자세한 내용은 [07_오픈소스_계산로직](07_오픈소스_계산로직/오픈소스_라이브러리.md)의 "lunar-flutter가 실제로 이 위험에 노출되는지 코드 레벨로 확정 확인" 절 참고. **2026-07-08 같은 세션 대안 발견**: 문제만 지적하지 않고 pub.dev에서 한국 전용 음력 변환 Dart 패키지를 찾아봄 — `klc`(KASI 기준, 1391~2050년 지원, 네트워크 불필요, MIT, `github.com/chunghha/dart_klc`)가 정확히 이 문제를 위해 존재함을 확인. 3년 전 마지막 게시로 오래됐지만 좁은 범위 유틸리티라 그 자체로 안정적일 가능성 있음(재확인 필요). `saju`의 `lunar` 의존성 대신 이 패키지로 교체하는 것이 한국 사용자에게 더 안전한 선택으로 보임.

**시장**: 한국 운세 시장 약 1.4조 원, 연 25% 성장. 포스텔러가 900만 가입자로 압도적 1위이며 이용자 80% 이상이 2030세대. 점신(1,500만 다운로드), 헬로우봇(다운로드 1위, 평점 4.7)이 뒤를 잇는 3강 구도.

**성공 공식**: 어려운 명리학 용어를 일상어/밈으로 재해석 + 캐릭터 기반 친근한 UI + 무료로 낮춘 진입장벽 + 유료 심화 리포트 전환 + SNS 공유 최적화.

**콘텐츠 톤 (실측)**: 포스텔러 인스타 최근 게시물 분석 결과, 명리학 개념을 "무자일주가 단톡에서 읽씹하는 이유"처럼 완전히 밈화해서 소비시키는 전략이 반응이 가장 좋음(좋아요 700~1,000+). 결과 화면 카피도 이 정도 캐주얼함을 목표로 삼을 만함.

**입력 설계**: 생년월일시·양력음력·성별이 필수, 출생지는 선택. "태어난 시간 모름" 케이스를 우아하게 처리하는 것이 이탈률에 직결.

## Agent-Reach 채널 현황 (2026-07-04 기준, `agent-reach doctor --json`; 2026-07-06에 실제 호출로 재검증)

이번 리서치에서 실제로 쓴 채널은 위 "주요 수집 채널" 열(웹 검색·GitHub·Instagram)이 전부지만, Agent-Reach 자체는 총 15개 채널을 지원한다. 아래는 현재 이 환경에서 각 채널이 실제로 작동하는지 점검한 결과이며, 향후 추가 리서치가 필요할 때 어떤 채널을 바로 쓸 수 있는지 참고용으로 남겨둔다.

**⚠️ 2026-07-06 실측 정정**: `agent-reach doctor`가 🟢로 보고하는 것은 "설정/바이너리가 설치돼 있다"는 뜻일 뿐, 실제 호출 성공을 보장하지 않는다. `opencli twitter search`, `opencli reddit search`를 이 자동화 세션에서 직접 실행해보니 **둘 다 `BROWSER_CONNECT`(Browser Bridge 확장 프로그램 미연결) 오류로 실패**했다 — OpenCLI 백엔드를 쓰는 채널(Twitter/X·Reddit·Facebook·Instagram·B站·샤오홍슈 전부)은 사람이 상시 켜둔 Chrome + OpenCLI 확장 프로그램이 필요한데, 이 무인 자동화 세션에는 그런 브라우저가 없다. 아래 표의 🟢는 "설정상 가능"일 뿐 "이 세션에서 바로 호출 가능"이 아니므로, 실제로 매번 쓸 수 있다고 확인된 채널은 **GitHub(`gh`)·전역 시맨틱 검색(Exa)·임의 웹페이지(Jina Reader)·YouTube(yt-dlp)·RSS 뿐**이다.

| 채널 | doctor 보고 상태 | 2026-07-06 실측 | 비고 |
|---|---|---|---|
| GitHub 저장소/코드 | 🟢 사용 가능 | ✅ 실제 작동 확인(이번 세션에서 다수 사용) | gh CLI, 읽기·검색·API 완전 지원 |
| 전역 시맨틱 검색 | 🟢 사용 가능 | ✅ 실제 작동 확인(이번 세션에서 다수 사용) | Exa(mcporter 경유), API 키 불필요 |
