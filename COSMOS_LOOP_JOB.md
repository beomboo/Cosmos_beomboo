# 오버나이트 자동화 루프 — 재사용 프롬프트

세션 기반 루프(`ScheduleWakeup`)에서 실제로 쓰이던 지시문을 다른 방식의 스케줄러(클라우드
cron `/schedule`, `CronCreate` 등)로 옮길 때 그대로 재사용할 수 있도록 정리한 문서다.
아래 "재사용 프롬프트" 블록은 자기완결적으로 작성되어 있어, 어떤 스케줄러에 넣더라도 매
실행이 이전 대화 맥락 없이 `PROJECT_ROUTER.md` 기록만으로 이어서 작업할 수 있다.

## 배경

- 이 프롬프트는 2026-07 초 COSMOS_BEOMBOO 오버나이트 자동화 세션에서 사용자 요청에
  따라 여러 차례 다듬어진 최종 형태다(1시간 간격, `develop` 브랜치, 사이클 후 자동 push,
  `PROGRESS.md` 대신 `PROJECT_ROUTER.md`만 사용 등).
- 세션 기반 루프(`ScheduleWakeup`)는 **대화 세션에 종속**된다 — 세션이 끊기면 예약도
  함께 사라진다. 세션과 무관하게 계속 돌게 하려면 클라우드 스케줄(`/schedule` →
  `CronCreate`)로 전환해야 하며, 이때 재예약 관련 문구만 교체하면 된다(아래 (8) 참고).

## 재사용 프롬프트

이전 실행의 대화 맥락이 아니라 PROJECT_ROUTER.md에 기록된 상태만으로 이어서 작업할 수 있어야
합니다. PROGRESS.md는 2026-07-05부로 더 이상 읽지도 쓰지도 않습니다 — 진행 상황 추적은
PROJECT_ROUTER.md의 기능별 상태(⚪/🟡/🟢)와 설명만으로 합니다.

(1) 시간대 확인: `TZ=Asia/Seoul date "+%H:%M"`로 현재 KST를 확인하고 아래 두 모드 중 하나로 분기한다.

  - 00:00~06:00 (리서치 보강 모드, 50분 간격으로 진행 — 아래 (8) 참고): saju-researcher
    에이전트(이 문서가 유일한 진입점)를 통해 Agent-Reach로 docs/research/ 내용을 보강한다.
    리서치 산출물은 사주팔자/궁합/MBTI/운세 4개 개념별 디렉터리(docs/research/ 아래)에
    정리하며, "정확·최신·중복 없음" 원칙을 따른다 — 사주팔자/궁합/MBTI(계산·콘텐츠 데이터)는
    개념당 파일 하나에 현재 가장 정확한 내용만 유지하고 새 조사 결과로 교체한다(날짜별 이력을
    쌓지 않는다), 운세(시장·트렌드)는 시점 자체가 의미 있어 관찰 기록 형태를 유지해도 된다.
    조사 중 UI/UX 목업(docs/mockups/*.html)을 구체적으로 개선할 근거를 찾으면 함께 개선한다.
    project/ 코드는 절대 건드리지 않으므로 빌드 검증은 생략한다.

  - 06:00~23:59 (개발 모드): 그때까지 보강된 docs/research/·docs/mockups/ 자료를 참고해
    project/lib/에서 실제 기능 개선을 진행한다. 표준 빌드 검증을 그대로 적용한다.
    (참고: 리서치 모드 종료 시각과 정확히 맞물리도록 06:00부터 시작한다 — 이전에는
    07:00부터였는데, 그러면 06:00~06:59가 어느 모드에도 안 걸리는 빈틈이 있었다.)

(2) 리서치 모드 데이터 수집 전략(2026-07-13, 사용자 요청 — 리서치 보강 모드에 한함): 매번
   무엇을 조사할지 즉흥적으로 정하지 않고, 아래 로테이션 표를 따른다. 00:00~06:00을 50분

   | 5 (04:10) | 사주팔자 | 공망·택일 보강, 또는 다른 경쟁앱 마케팅 자료에서 아직 못 본 새 개념이 있는지 재탐색 |

   | 7 (05:50) | **사주팔자**(마무리) | 그날 밤의 마지막 슬롯 — `사주팔자/` 데이터베이스를 최우선으로 점검·보완하고 마무리한다(2026-07-13, 사용자 요청: "사주팔자 데이터베이스는 개발 단계에서 적용돼 사용자에게 일관되고 정확한 정보를 전달해야 한다" — 궁합·MBTI·운세보다 정확성 요구가 가장 높은 핵심 데이터베이스이므로 매일 밤 마지막에 한 번 더 정합성을 확인). 개념 파일 사이에 서로 모순되는 서술은 없는지, 조견표(십이운성·신살 등)에 오탈자·불일치는 없는지, "정확·최신·중복 없음" 원칙에 어긋나는 부분이 있는지 확인 |

   각 파일 제목 바로 아래에는 `_최종 확인: YYYY-MM-DD_` 한 줄이 있다(saju-researcher 에이전트
   정의 참고) — 이미 그 폴더/항목이 최근에(이 날짜로 확인) 다뤄졌다면, 표의 다음 순번 항목으로
   대체해도 된다. 로테이션은 "매번 다른 곳을 보게 하는" 가이드이지 절대 규칙은 아니다. 조사
   결과는 `docs/research/`의 해당 개념별 파일에 "정확·최신·중복 없음" 원칙(위 (1) 참고)대로
   반영하고, 파일을 고쳤다면 이 날짜 마커도 오늘 날짜로 갱신한다.

   `saju-researcher` 에이전트를 호출할 때는 이번 슬롯에서 계산한 대상·조사 방향을 그대로
   에이전트 프롬프트에 적어서 전달한다 — 에이전트 자신은 이전 사이클 맥락이 없으므로, 무엇을
   조사할지는 이 문서를 읽는 쪽(메인 세션)이 매번 계산해서 넘겨줘야 한다.

## 스케줄러 전환 시 참고할 점

- 세션 기반(`ScheduleWakeup`)은 최대 1시간 간격 제한이 있고, 세션이 끊기면 예약도 사라진다.
- 클라우드 기반(`/schedule` → `CronCreate`)으로 옮기면 세션과 무관하게 계속 돌지만, 사이클
  안에서 하던 "매번 다른 각도로 탐색"·"가이드레일 준수"·"PROJECT_ROUTER.md만 갱신" 같은
  판단은 매 실행마다 위 프롬프트 텍스트 하나에 전적으로 의존하게 되므로, 그대로 cron
  프롬프트에 넣으면 지금까지의 동작을 재현할 수 있다.
- `.claude/settings.json` 관련 예외 처리((7)번)는 스케줄러 종류와 무관하게 항상 적용해야
  하는 규칙이다(권한 파일 자동 커밋 방지).
